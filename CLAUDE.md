## 二、红线（违反即打回，不必讨论）

1. **官方净值走采集器 KV（唯一来源），盘中数据走 `DATA_MODE` 双模**（[D-023](docs/DECISIONS.md#d-023) G 节）。
   **`DATA_MODE` 不再管官方净值**——`fetchOfficialData` 只读 `NAV_BASE/v1/nav/today`，
   拿不到就走 `officialBatchCache` 上次好数据。**不许再加"直连兜底"**：浏览器侧东财恒被
   `ErrCode:61136` 拦、腾讯只有一路，兜不出第二个源，加回来只是把已收敛的取数路径重新摊开。
   盘中 / 周末 / 节假日没有当日记录时，读端点回退 `nav:latest`，返回的 `date` 是**记录自带日期**，
   两类口径（红线 #6）据它照旧判新旧，不需要感知这层回退。

   以下双模只适用于**盘中数据（估算、指数）**：`config.js` 的 `DATA_MODE` 切换两条链路——
   - `"gateway"`：走 `API_BASE` 三个端点，主备切换 + KV last-known-good 全在 `workers/fund-market-api/` 网关内。
   - `"direct"`：浏览器直连。**估算**腾讯 `TX_BASE`（`_fetchTencentFunds`，只产出 estimate 块）；
     **指数**单源腾讯（`_fetchIndexGroupTencent`）——新浪备源因无 CORS 头在浏览器侧不可用，
     已删，见 [D-020](docs/DECISIONS.md#d-020)。
   改盘中数据逻辑只在这两套分支内，不得到处加 fetch。
   **选新数据源前先验 CORS**：浏览器直连要求对方返回 `Access-Control-Allow-Origin`，
   且 `Referer` / `User-Agent` 属 fetch 禁止修改头，靠伪造它们绕防盗链在浏览器里一律无效。
2. **行情数据禁止因单次失败清空**：拿不到新数据时退回上次好数据并标记为陈旧，绝不把已有数据冲成空。
   保护落在**网关侧**（对所有设备生效），不是浏览器 localStorage。详见 [D-001](docs/DECISIONS.md)。
3. **`js/` 无文件数上限**（2026-07-21 解除，原为「上限 10 个，已满」）。但**新增文件仍需先问**（见 #10），
   且必须落在既有分层里、在 `index.html` 按依赖顺序加载。解除上限不等于鼓励开新文件——
   **先问「这个函数是不是本来就该属于某个既有层」**：`applyProxyEstimates` 一度因"ui 不该算业务"
   要单开一层，实际它是纯函数、本就属于 engine，放对地方后问题自然消失（D-022）。
4. **业务数字只在 `config.js`**，不许散落。
5. **配置写入必经 store 封装**，每个封装自己调 `bumpConfigVer()`，不许依赖调用方顺带调了别的。
6. **净值判据分两类口径，不许混用**（[D-019](docs/DECISIONS.md#d-019)）：
   - **市值口径**（持仓值多少钱）：`data.js` 的 `getNavByCode`，判据 `f.offVal && (!estD || offD >= estD)`，
     要**最新可得**净值，哪天的都对。
   - **收益口径**（今天赚了多少）：`engine.js` 的 `calcTodayProfit` 与 `interact.js` 的 `profitMap`，
     **必须是今日数据**（`offD===today` 或 `estD===today`，官方优先），非交易日才回退最近可得。
     **这两处必须逐字同步**——只改一处就会重现「顶部归零、抽屉却有收益」（D-010 修过一次）。
     基准代理不在这里分叉：它在 `refreshData` 入库前就填进 `results` 的估算位了（D-022），
     两处看到的都是同一条净值链，**不要再往这两个函数里加取值分支**。
   - 曾经写成"三处逐字一致"，把两种口径捆在一起，导致盘中估算断供时把上一交易日的涨跌
     当成今日收益显示。
7. **engine 是纯函数**，外部依赖全部参数传入。
8. **改后代码量不许增加**：改根源不包裹，删死变量僵尸函数。
   界定（2026-07-21 补）：约束的是**包裹式膨胀**——加 if 绕开症状、留"以后可能用得上"的死变量、
   保留不工作的僵尸函数。**新增真实能力可以净增行数，但要在决策条目里如实报数**，
   不许像 D-020 初版那样在风险表里写"总量持平"而实际净增 300+ 行。
   自查三问：删掉的是不是比留下的多？留下的每个函数是不是真的会被调用？行数增长换来了什么能力？
9. **不擅自补功能，但鼓励提议**：审查只报问题，觉得缺功能 → **说出来、给建议、等裁决**，不动手。
   界定（2026-07-21 补）：约束的是"AI 自己起意加东西"，不是"闭嘴"。
   **执行已批方案时发现方案本身有缺口，属于提议范畴——提出来，别默默补也别默默跳过。**
   用户明确表示：提议比不提议更受欢迎，裁决权在用户，提议权在你。
10. **结构性改动先问**：目录、文件清单、跨层方案、删疑似有用的函数。

## 三、防信息丢失机制（这套的存在理由，务必遵守）

**已经发生过的事故**：2026-06-19 建立的"行情数据 last-known-good 保护"只活在代码里，从未写进文档；
2026-07-20 做网关迁移时，AI 读了文档里"整组失败即不可用"的表述，照着执行，把保护拆了，
导致收盘后一次刷新把全天数据冲成空白。**根因不是记性差，是判断依据没有被固化。**

因此立下三条：

### 3.1 决策日志只增不改

`docs/DECISIONS.md` 是**追加式**的。**任何"删除、替换、绕过既有机制"的改动，先在其中追加条目，再动代码。**
条目写清：背景、决策、**理由**、代价、代码位置、**须同步修订的正文章节**、取代了哪条。
理由那栏是重点——代码能表达"做了什么"，表达不了"为什么这么做、为什么不那么做"。

**「须同步修订的正文章节」必填**，是堵正文腐烂的触发点。写决策是"追加"，成本低、有仪式、这里反复强调；
回头修正文是"修改"，成本高、没有触发点、没人催。久而久之决策日志很健康，而 `01`–`05` 正文在烂——
D-006 拆掉 PE 双路验证层时条目写得完整，但 `01` §1.2 描述的那套机制自那时起就不存在了，
一路烂到 2026-07-21 才被发现（见 D-016）。填了这栏，改完代码才有回头改正文的钩子。

**什么不需要写决策**：判据是**理由能否从代码本身复原**。能 → 只写代码注释，不占条目。
对照：`totalFriction:0`（D-004）必须写，代码看着就像 bug；in-flight 去重（D-011）本可不写，代码意图自明。
**条目越滥，索引表越不可信，跳读就失效了**——那才是决策日志真正会死的方式，不是文件变大。

**分卷不算改**：超过约 800 行时，把状态为 `已被取代 / 已失效` 的条目整体移入
`docs/archive/DECISIONS-<年份>.md`，索引表保留该行并注明去向。**迁移不是删除**，与 3.2 同一套逻辑，
只增不改完好无损。（2026-07-21 已触发一次，D-008 迁入 `docs/archive/DECISIONS-2026.md`。）
**注意该触发条件目前近乎失效**：20 条里 19 条是「生效中」，唯一可迁的只有 D-008，
迁完仍有 900+ 行。真要控制体积得另立判据（按年份归档、或长条目拆附录），届时补一条决策。

### 3.2 文档不许原地删

某节被整体取代时，把旧版移进 `docs/archive/<原文件名>-<日期>.md`，不要直接删。
`docs/archive/` 是给未来的 AI 看的考古层，不参与日常阅读。

### 3.3 代码里的反直觉设计必须双写

凡是"看起来像 bug、其实是有意为之"的地方（如 `totalFriction:0`、`xqNav` 口径、`getDynamicTarget` 边界兜底），
代码注释和 `docs/03-实现约束.md` 两边都要有，且注明**为什么**。只写一边等于没写。

> **给 AI 的话**：你读到的文档可能不完整，代码里可能有文档没记的设计。
> **发现代码与文档冲突时，默认代码是对的、文档是漏的**——先问，不要按文档去"修正"代码。
> 上面那起事故就是反着来造成的。

## 四、项目速览

纯原生前端，零构建，零框架，唯一外部库 CDN 的 `Sortable.js`。

```
fund-monitor/
├── CLAUDE.md                 # 本文件（索引）
├── docs/                     # 规范全文 + 决策日志 + 归档
├── index.html · favicon.png · CNAME
├── css/style.css
├── js/                       # 前端运行时，当前 10 个文件（数量上限已解除，见 D-021）
│   config · logger · store · data · engine · ui · ui-holding · ui-pe · interact · main
├── workers/
│   ├── fund-market-api/      # 市场数据网关（Cloudflare Pages Functions，当前休眠）
│   ├── fund-nav-collector/   # 官方净值夜间采集器（Cloudflare Worker + Cron + KV，D-023）
│   └── pe-night-trigger/     # 夜间引擎触发器（Cloudflare Worker）
└── automation/               # 夜间 PE 数据引擎（Python + GitHub Actions）
```

**官方净值夜间采集器（`workers/fund-nav-collector/`，详见 D-023）**：带 Cron 的 Worker，
19:00–23:00 北京每分钟一跳，**并行**打东财与腾讯，逐只只收当日净值、**先到先得**记账
（`src` 谁给的 / `at` 何时给的，写入后不可变），写 KV。全部到齐即早退。
⚠️ **读写分在两个部署里**：Worker 只写；前端读的 `/v1/nav/today` 挂在**网关** `router.mjs`，
故 `NAV_BASE = API_BASE`。成因是 Worker 拿不到大陆可达域名（`bailuzun.com` 不在 CF zone，
`*.workers.dev` 大陆不可达），详见 `workers/fund-nav-collector/README.md`「读写分离」节。
目标基金列表读 Gist `fm_config.json` 的 `f`，跟随看板增删，**用户零操作**。
表头与手机卡片的标签「腾讯 2」就是它算出来的：今晚最早抢到的那一路 + 它抢到的只数。
**Worker 未部署时看板行为与接入前完全一致**——采集器只是 direct 的子路径。

**数据源双模（`DATA_MODE`，详见 `docs/DECISIONS.md` D-013、D-020）**：
`config.js` 一个常量控制全站走网关还是浏览器直连。`"direct"` 时分两路取数——
**官方净值主东财**（`fundmobapi.eastmoney.com`，JSON）**备腾讯**（`jj{code}`，GBK 须 `TextDecoder("gbk")` 解码）；
**指数单源腾讯**（`qt.gtimg.cn`）——曾接新浪备源，因其不返回 CORS 头浏览器取不到，代码已删（D-020 代价栏）。
`workers/` 网关此时不被前端使用，仅作为备选保留。`DATA_MODE` 当前实际值为 `"direct"`。

**基准代理（`DATA_SOURCE_SWITCH`，详见 D-020、D-022）**：盘中估算源全部失效后，
用「产品自定义基准权重 × 实时指数」推日内方向。**回填发生在 `refreshData` 入库前**（D-022），
于是权益%、持仓总额、最新收益、估算列、增权/降权预案读的是**同一条净值链**；
官方净值当晚落地后由 `getNavByCode` 的 `offD >= estD` 一只一只替换回官方。
指数每 10 秒一跳都会触发 `reapplyProxyEstimates` 重算。
权重表 `BENCHMARK_PROXY` 字段为 `legs`（权益腿 / 纯债的国债腿共存），须随季报手工校核。

**PE 锚定双路（`PE_ANCHOR`，详见 D-017）**：`config.js` 一个常量决定实时 PE 走哪条——
`"mcap"` 总市值路（默认，D-006 实测误差 0.41pp）、`"price"` 点位路（1.31pp，但成分调整日更可靠）。
**全站经 `getAnchorPE` 取值，禁止直接调 `getEnginePE` / `getEnginePE1`**；另一条由 `getRefPE`
自动降为 PE 栏小字。曾因两处各自直调而 PE 栏与持仓抽屉判定不一致。

**估算持久化（D-014 定结构，D-018 修实现）**：直连模式盘后腾讯返回 0 → 回退 **localStorage 缓存**
→ 再回退 **Gist `fm_est.json`**（跨设备兜底，收盘后推一次，读侧 30 分钟节流）。
三层链：fresh → localStorage → Gist → unavailable。缓存**不存日期**，新旧一律按每条自带的
`estimateAt` 判，非当日即挂「陈旧」小字（卡片与表格两个视图都挂）。网关模式不受影响。

加载顺序：`config → logger → store → data → engine → ui → ui-holding → ui-pe → interact → main`

分层依赖单向向下，下层绝不调上层：

```
config → store → data ─┐
                        ├→ ui → interact → main
              engine ──┘
```

**`js/` 的文件数上限已于 2026-07-21 解除（D-021）**，新增仍需先问；`workers/` 与 `automation/` 从来不占配额。
