# 第三篇 · 实现约束

## 3.1 localStorage 封装

interact 层和 ui 层**禁止直接调用 `localStorage`**，所有读写通过 store 封装。

| 数据类型  | 读                                                           | 写                   | 清                    |
| --------- | ------------------------------------------------------------ | -------------------- | --------------------- |
| 基金列表  | `loadFunds()`                                                | `saveFunds()` ⚙      | —                     |
| PE 定锚   | `loadPe()`                                                   | `savePe()`           | —                     |
| 持仓数据  | `loadHoldings()` / `loadHoldingsEquity()` / `loadShortNames()` | `saveHoldingsData()` ⚙ | —                   |
| 降权预案  | `loadSellPlan()`                                             | `saveSellPlan()` ⚙   | —                     |
| 优先卖出  | `loadPrioritySell()`                                         | `savePrioritySell()` ⚙ | `clearPrioritySell()` ⚙ |
| 配置版本  | `loadConfigVer()`                                            | `bumpConfigVer()`    | —                     |
| Gist 配置 | `loadGistConfig()`                                           | `saveGistConfig()`   | `clearGistConfig()`   |

> ⚙ = 该封装内置 `bumpConfigVer()`。**配置内容（f/h/s/pr）的每一个写入封装都必须自己自增版本**，
> 不许依赖「调用方会顺带调另一个会自增的函数」——`saveSellPlan` 就曾因唯一调用点紧跟在
> `saveHoldingsData` 之后而漏掉，属于随时会被下一次改动踩响的哑雷，2026-07-19 已补。

**授权例外**

- `syncCloud` push 路径直接读 `localStorage.getItem(STORE_HOLDINGS)` 跳过缓存，保证推送实时性。
- `importSnapshot` 内部直接写 `f/h/s/pr` 与配置版本号（不经 `saveSellPlan` 等封装，避免重复 `fmLog`），最后统一广播 `FUNDS` + `LOCAL_CONFIG`，避免双重渲染。

## 3.2 资金规则 → 代码映射

业务规则（1.5–1.6）落到 engine 的对应关系，改动须对照业务验证：

- **增权 `calcBuyPlanDraft`**：`buyAmt` = 缺口现金；`allocA500C = min(buyAmt, a500cRoom)`，`a500cRoom` 受 `LIMIT_A500C` 约束；`allocZZ500C` = 溢出，按 `isZZ500Product` 找承接品种。
  - ⚠️ `sharesA500C`/`sharesZZ500C` **除以兴全净值 `xqNav`**，因为业务关注的是「为各买入桶需**卖出的兴全份额**」，不是买入标的份额——**有意设计，勿改成除以 a500cNav**。
  - ⚠️ `totalFriction: 0` **有意设计**：增权标的为纯债（卖出端）和 C 类（买入端），均零摩擦，用户自行掌握。
- **降权 `calcSellExecutionDraft`**：三层优先级——中证500（`isZZ500Product`）→ `priorityCode`（用户标记）→ 其余按 `ratios` 比例。仅 `equity>0` 的品种参与。
- **档位目标 `getDynamicTarget(mode, bucketStr)`**：`buy` 取下一档、`sell` 取上一档、`neutral` 取本档。**边界兜底**：最低档增权或最高档降权时返回本档 target，业务合理，不是 bug。
- **实时 PE `getCurrentPE(peData, engineResult)`**：**不做插值**。`peData` 只承载 `bucketStr`，本函数只用它算跨档线（buyPct = lo − BUFFER_ZONE，sellPct = hi + BUFFER_ZONE）；PE 数值整个来自 `engineResult`（旁路引擎百分位）。引擎未就绪时 `value` 为 `undefined`，由 `ui-pe.js` 的 `Number.isFinite` 守卫落入「等待数据」。
  - ⚠️ 早年的「夜间录入三个点位 + `priceBuy/priceAnchor/priceSell` 分段线性插值」**已不存在**，见 [D-016](DECISIONS.md#d-016)。旧文档归档在 `docs/archive/`。
- **PE 锚定路径 `getAnchorPE` / `getRefPE`（[D-017](DECISIONS.md#d-017)）**：`config.js` 的 `PE_ANCHOR` 决定主路径——`"mcap"` 走 `getEnginePE1`（总市值路，默认），`"price"` 走 `getEnginePE`（点位路）。`getRefPE` 恒返回**另一条**，供 PE 栏小字并列展示。
  - ⚠️ **新增任何 PE 调用点一律走 `getAnchorPE`，禁止直接调 `getEnginePE` / `getEnginePE1`。** 2026-07-21 之前 PE 栏走 1.0、持仓抽屉走 2.0，同一份持仓两处 `isEquityWrongDir` 判定可能相反，PE 贴近 `PE_HIGH_THRESHOLD`(65) 时尤其容易撞上。
  - CSS 类名 `pe-bypass2` 是 `PE_ANCHOR="mcap"` 时代的遗留（那时小字恒为 2.0）；翻成 `"price"` 后小字变 1.0，类名语义反转。**已知，纯命名问题，不影响渲染，故意不动。**
- **方向异常 `isEquityWrongDir`**：高估区（PE ≥ `PE_HIGH_THRESHOLD`）权益偏低超 `EQUITY_DEV_LIMIT`，或低估区权益偏高超阈值。

**业务数字一律在 config.js**（顶层裸常量，按 §1–§10 分节；原 `SYS_CONFIG` 聚合已删，见 D-020）：费率 `FEE`、缓冲带 `BUFFER_ZONE`、阈值 `PE_HIGH_THRESHOLD`/`EQUITY_DEV_LIMIT`、代码 `CODE_XQ`/`CODE_A500`/`IDX_PE`、上限 `LIMIT_A500C`、时点 `T_*`/`T_OFF_UPDATE`、默认档位 `DEFAULT_BUCKET`、12 档表 `PE_EQUITY_TABLE`、模式开关 `DATA_MODE`/`PE_ANCHOR`、估算缓存 `EST_CACHE_MAX_AGE`/`EST_GIST_READ_THROTTLE`。新增业务数字不得散落他处。

## 3.3 engine 净值/档位取值规则

**净值选取分两类口径**（[D-019](DECISIONS.md#d-019)）。曾经写成"三处逐字一致"，把两种口径捆在一起，
结果盘中估算断供时把上一交易日的涨跌当成今日收益发了出去。**先分清在算什么，再决定用哪条判据。**

| 口径 | 位置 | 判据 | 为什么 |
| --- | --- | --- | --- |
| **市值**<br>持仓值多少钱 | `data.js` `getNavByCode` | `f.offVal && (!estD \|\| offD >= estD)` | 要**最新可得**净值，哪天的都对。加日期闸门反而会让持仓总额在盘中归零。D-022 后估算位盘中恒有代理值，故盘中取代理、官方落地即自动切回，权益%/总额因此能跟随大盘 |
| **收益**<br>今天赚了多少 | `engine.js` `calcTodayProfit`<br>`interact.js` `profitMap` | **今日数据**：`offD===today` 或 `estD===today`，同为今日时官方优先；非交易日（`WEEKEND`/`BEFORE_PRE`）才回退最近可得 | 盘中官方永远停在昨天，不加闸门就会把昨天的涨跌当成今天的 |

⚠️ **收益口径那两处必须逐字同步。** 顶部与抽屉是同一口径的两个出口，只改一处就会重现
「顶部今日涨跌归零，抽屉里却照常有收益，两者对不上」——[D-010](DECISIONS.md#d-010) 修过一次。

⚠️ **基准代理不在这两处分叉**（[D-022](DECISIONS.md#d-022)）。代理值由 `refreshData` 在
`setLastResults` 前填进 `results` 的估算位，两处看到的都是同一条净值链，各自不带代理分支。
D-020 曾在这两个函数里各加一段代理兜底，D-022 上移后它们永不进入、已按红线 #8 删除。
**不要再往这两个函数里加取值分支**——要改代理行为，改 `applyProxyEstimates` 一处即可。

⚠️ **市值口径那条的 `!estD ||` 半边不能省。** 曾写成 `estD && offD && offD >= estD`，要求先有估算才肯用官方；
估算整组不可用时所有 `estVal` 同时为 null，持仓市值便整体算不出来。2026-07-19 已修正。

**"已更新"徽标**：由 `calcTodayProfit` 的 `isRealUpdate` 驱动 `allUpdated` → 顶部徽标。
[D-010](DECISIONS.md#d-010) 当初把它与净值选取拆开，理由是"数值可以放宽（fallback），状态必须严格"。
- ⚠️ **代理不算"已更新"**（[D-022](DECISIONS.md#d-022)）：`isRealUpdate = isOffToday || (isEstToday && estSource !== "proxy")`，
  与取值用的 `isFundUpdated`（含代理）刻意不同。**看着像重复判断，其实是 D-010 那条原则的落地**——
  代理是拿昨日净值乘指数推的，一条真数据都没到；若徽标沿用 `isFundUpdated`，
  估算源全死的日子里它会天天亮着，等于永久说谎。**别"顺手"把这两个合并回一个变量。**
**[D-019](DECISIONS.md#d-019) 收紧了前半句**：收益口径的数值现在同样要求今日数据，fallback 的范围
从"任意可得"缩小为"今日数据，官方优先"，仅非交易日例外。徽标判据本身未变，D-010 的拆分结构仍然成立——
也正是靠这个拆分（徽标灭、数字却照发）才定位到 D-019 那个 bug。

**顶部「最新收益」的显示**（[D-019](DECISIONS.md#d-019)）：`calcTodayProfit` 返回 `isWaitingForData`，
为真时顶部显示 `-`。三种情形——盘前集合竞价、有持仓但无一只拿得出今日数据、整组请求失败。
**有持仓就是有持仓，不因请求失败而变空白**：空白看起来像"没有持仓"，正确表达是"有持仓但此刻不知道"。

所有品种（含纯债）正常情况下都有盘中估算数据（2026-07-17 起估算源断供，见 [D-015](DECISIONS.md#d-015)）。

**官方净值 TTL（`fetchOfficialData`）**：官方数据按基金列表整体缓存；19:30（`T_OFF_UPDATE`）前 1 小时 / 后 5 分钟 / 已是当日数据或周末 12 小时。避免按基金分别缓存造成主源与备用源混用。

**估算持久化（直连模式专用，[D-014](DECISIONS.md#d-014) 定结构、[D-018](DECISIONS.md#d-018) 修实现）**：
- **回退链**：`fetchEstimates` 直连模式 — fresh(腾讯) → localStorage → Gist → unavailable。
- **localStorage** `fm_est_cache_v1`：读写一律经 store 的 `saveEstCache`/`loadEstCache`/`loadEstCacheEntry`，data 层禁止裸调 `localStorage`。
  - ⚠️ **结构是 `{ ts, data }`，刻意不存日期。** 旧实现存了 `date` 且 Gist 回退时无条件盖成今天，把隔周数据洗成「当日」。没有这个字段，那个 bug 就无处可藏——**新增字段前先想清楚它会不会被写错**。
  - `ts` 只作整份丢弃的硬上限（`EST_CACHE_MAX_AGE`，7 天），**不参与新旧判断**。
- **新旧判断与标签**：由每条估算自带的 `estimateAt` 决定，不按来源判。`ui.js` 的 `srcTag(source, dataDate, today)`——
  源名（`tencent`/`eastmoney`/`sina`/`backup`）直接出名字，主源也出声；`stale`→「陈旧」；
  `cached`/`gist` 仅当 `dataDate !== today` 才「陈旧」（收盘后沿用当日 14:59 那笔仍是当日数据，标陈旧反而不准）；
  `proxy`→「代理」（`.is-alt` 告警底）。见 [D-020](DECISIONS.md#d-020) C/D 节。
  - ⚠️ **标签挂在表头，不在每行**：`renderSourceTags` 写 `<th>` 里的 `thEstSrc`/`thOffSrc`，每列只出现一次。
    官方列所有基金必然同源（整组取源，见 [D-002](DECISIONS.md#d-002)），逐行重复是噪音。
    卡片视图不挂——手机收窄本就该少字。**改此处别退回"每张卡各挂一个"的旧写法。**
  - ⚠️ **估算列可能混源，取标签时真实源优先**：代理是逐只让路的，某只估算源复活时其余仍是代理。
    `renderSourceTags` 的 `pick` 分两轮——先找非 `proxy` 的源，找不到才退回 `proxy`。
    **顺序反过来会把"恢复"整个盖住**：排在前面的代理会一直占着表头，而这一格变字
    是察觉估算源复活的唯一信号（那是个没有通知的事件）。
- **Gist** `fm_est.json`：`pushEstToGist()` 只负责推，**时机由 `interact.js` 的 `refreshData` 在 `mkt === "POST_MARKET"` 判定**——`getMarketState` 属 engine，与 data 平级互不依赖，判断不能放 data 层。
  - 日期标记 `fm_est_gist_date_v1` 在**推送成功后**才落；先落会让一次失败吞掉当天仅有的机会。
- **Gist 读侧节流** `EST_GIST_READ_THROTTLE`（30 分钟），**成败均推进时间戳**。估算源断供时 `fm_est.json` 可能长期不存在，只在成功时节流等于没节流，会退化成每 60 秒一次注定 404 的认证请求。
- **代理模式不变**：`DATA_MODE="gateway"` 时 `needEstimate` 原策略保留，不触发估算缓存。

## 3.4 染色体系

| 变量       | 唯一语义                       |
| ---------- | ------------------------------ |
| `--buy`    | 增权方向                       |
| `--sell`   | 降权方向、优先卖出标记         |
| `--up`     | 行情上涨                       |
| `--dn`     | 行情下跌、完成/已更新状态      |
| `--warn`   | 方向异常（`wrongDir` 为 true） |
| `--t1`     | 已对齐（偏离 < 1%）的权益值    |
| `--t2`     | 偏离 < 1% 的偏差数字、辅助信息 |
| `--t3`     | 无数据、占位文字               |
| `--accent` | 持仓总额、主操作按钮           |

**权益/偏离值染色**（优先级从高到低）：

| 状态     | 条件             | 权益色   | 偏离色   |
| -------- | ---------------- | -------- | -------- |
| 无数据   | `eqData == null` | `--t3`   | —        |
| 方向异常 | `wrongDir`       | `--warn` | `--warn` |
| 已对齐   | `\|diff\| < 1%`  | `--t1`   | `--t2`   |
| 偏高降权 | `diff >= 1%`     | `--sell` | `--sell` |
| 偏低增权 | `diff <= -1%`    | `--buy`  | `--buy`  |

## 3.5 样式约定

- **字体**：数字加 `class="num"`（`--f-num`），中文用 `--f-zh`，禁止遗漏。
- **颜色**：全部用 CSS 变量，禁止硬编码 hex/rgb/具名色。唯一例外：遮罩层 `rgba(0,0,0,0.6)`。
- **字重**：只用 400 / 500 / 600，禁止 700。
- **动态 `<input>`**：必须声明字体、背景、边框、文字颜色（不继承父级）。
- **网格边框**：容器 `gap:1px` + 边框底色，单元格设卡片底色，天然细线；禁止 `:nth-child` + `border` 覆写。
- **抽屉 Tab**：`tabStyle` 用 `var(--bg3)`；增权 Tab 用 `var(--buy-bg)`、降权用 `var(--sell-bg)`；卡片体统一 `var(--bg3)`，方向感由数字颜色和边框传达。
- **护眼模式**：低饱和度护眼调色盘，`<meta name="darkreader-lock" />` 防外部劫持；除 `--up/--dn` 外禁止高亮纯白。

## 3.6 ui 三文件关键约定

- **DOM 缓存**：高频静态节点通过 `_getEl(id)`，禁止在渲染循环内直接 `getElementById`。
- **`_peDOMReady` flag**：控制 `_peDOM` 一次性初始化，禁止用 `!_peDOM.display` 判断（display 为 null 时误判）。
- **`UI_render*` 工厂**：只返回 HTML 字符串，DOM 写入由 interact 负责。
- **`UI_buildSummaryHtml` 签名**：`(currentPE, eqData, currentEqVal, eqCol, targetNeutralNum, inDrawer=false)`，`inDrawer=true` 自动输出抽屉样式。
- **`UI_buildHoldingPlanHtml` 签名**：`(activeProds, targetEqNeutral, eqData, buyDraft, sellDraft)`。`eqData` 由 `liveUpdateHoldingPlan` 计算后传入，ui 层禁止自行调 `calcCurrentEquity`。

> **唯一例外 · PE 栏只读展示**：`ui-pe.js` 的 `updatePeBar` 直接调用 `calcCurrentEquity` /
> `getDynamicTarget` / `isEquityWrongDir` / `getBoundaryPrices`，与本节及 [02-系统架构.md](02-系统架构.md)
> §2.2「ui 层禁止业务计算」相抵触。
>
> **为什么开这个口子**：PE 栏挂在 `INDICES` 频道上，每 10 秒随指数刷新；若按常规交由 interact
> 预算再传入，就要为这条 10 秒链路单开一个调度函数并改动 Observer 接线，**代价大于收益**。
>
> **边界**：仅限只读展示，**不得写 store、不得触发同步**。这是白名单，不是先例——
> 其他 ui 文件仍然禁止业务计算。
- **`renderTodayProfit` 签名**：`(results, holdings, activeProds, mktState, todayStr)`，
  `holdings` 与 `activeProds` 由调用方传入。不需要 `indicesMap`——代理值已在 `results` 里（D-022）。
- **给视图新增跨频道依赖时，必须接上那条频道的重画线**（D-020 B' → [D-022](DECISIONS.md#d-022) F）：
  `INDICES` 频道除 `UI_updateIndices` 外还调 `reapplyProxyEstimates`，后者经 `setLastResults`
  广播 FUNDS 触发重画。漏接的表现不是报错，而是净值先于指数到货的那一轮渲染完就永久静止
  （顶部卡在 `-`、权益%冻结），要等 60 秒后的下一次净值刷新才恢复。
- **`applyProxyEstimates` 必须幂等**（D-022）：跳过条件是"当日已有**真**估算"
  （`estSource !== "proxy"`），不是"当日已有估算"。写成后者的话，自产的代理条目会挡住自己，
  指数再涨也不重算——症状是权益%在第一次回填后就再不动了。
- **卡片左侧色条**：`renderCards` 用 `getActivePct()` 定 `up-card`/`down-card`。折叠态卡片上只露一个数字，
  而那个数字由 `miniMode`（0=估算 / 1=官方 / 2=并列）决定，两者口径可能不同——**已知不一致，待定**，
  改动前先与用户确认，不要自行对齐。

**3.7 的 logger.js 曾是"破例的第 10 个文件"**；文件数上限已于 2026-07-21 解除（D-021），该说法只余历史意义，见 2.1。

## 3.7 日志诊断系统（logger.js）

长期保留（当年作为"破例的第 10 个文件"引入，上限已解除，见 D-021）。记录本地写入与云端推送（时间戳、客户端标识、函数名、数据快照），写入主 Gist 的 `fm_log.json`，最多 200 条滚动覆盖（约 10 天）。未配置 Gist 时静默跳过。

**配置**：与主 Gist 共用 `GistID,Token`，首次使用前须在主 Gist 手动添加 `fm_log.json`，初始内容 `[]`。

**接入点（10 处）**：store.js 的 `saveFunds`、`savePe`、`saveHoldingsData`、`saveSellPlan`、`savePrioritySell`、`clearPrioritySell`、`importSnapshot`（记录 `v/f/h/s/pr`）、`importPeSnapshot`；interact.js 的 `syncCloud_push_pe`、`syncCloud_push_config`（含版本号 `v`）。

**已知限制**：`fmLog` 为非原子的「GET 整份 → push → PATCH 整份」，同瞬两个 `fmLog` 会竞争同一文件、后者覆盖前者。配置版本机制落地后 import 已幂等、并发写日志极少，实务无影响；若需绝对可靠可改串行队列，非必需。

**停用**：删除 `logger.js`、删除 `index.html` 中的 script 标签、删除 store.js 与 interact.js 中所有 `fmLog(...)` 调用行。
