## D-019 · 收益口径加日期闸门，「今日收益」更名「最新收益」；红线 #6 改为两类口径

**状态**：生效中
**日期**：2026-07-21
**关联**：[D-010](#d-010)（徽标与净值判据拆分，本条收紧其"数值可放宽"那半）、[D-015](#d-015)（估算断供）、[D-018](#d-018)

### 背景

用户 2026-07-21 盘中报告：估算数据为空，顶部却显示「今日收益 +160.13」。

复现（今天周二 07-21，官方停在 07-20，估算源断供）：

| 场景 | 官方 | 估算 | 显示 | 实际是什么 |
| --- | --- | --- | --- | --- |
| 正常盘中 | 07-20 | **07-21** | +90.00 | 今天的 ✓ |
| **估算断供** | 07-20 | 07-20 或无 | **+1010.00** | **周一的收益** ✗ |
| 盘后官方已出 | **07-21** | 07-21 | +190.00 | 今天的 ✓ |

**根因**：`calcTodayProfit` 取 nav/pct 的判据是 `f.offVal && (!estD || offD >= estD)`，
**整条路径没有任何"这数据是不是今天的"闸门**。估算缺席时它就退到官方，而盘中官方永远是昨天的，
于是把上一交易日的涨跌当成今日收益显示出来。

`isFundUpdated`（[D-010](#d-010) 拆出的徽标判据）此时是灭的，**徽标诚实、数字不诚实**——
这比两者都错更糟，因为红色大字比灰色小徽标显眼得多，用户先看到的是错的那个。

**这个 bug 早于 D-018，2026-07-21 的全量审查逐行读过 `calcTodayProfit` 却没发现**：
当时注意力全在 D-010 的徽标拆分上，默认"数值那条路已经被 D-010 审过了"，
没有独立验证它。**审查时对已有决策背书的代码要格外小心——决策记的是当初的意图，不是现在的行为。**

### 决策

1. **收益口径加日期闸门**：只有 `offD === today` 或 `estD === today` 的基金才参与收益计算；
   两者皆非今日则该只不参与。同为今日时**官方优先于估算**。
2. **无今日数据时的显示**，按市场状态分两种：
   - `WEEKEND` / `BEFORE_PRE`（今天本就不该有数据）→ 回退到最近可得数据，照常显示。
   - 其余时段（今天该有却没有，是异常）→ 显示 `-`，复用既有的 `isWaitingForOpen` 分支。
3. **「今日收益」更名「最新收益」**（`index.html` 的 `今日涨跌` 同步改）。非交易日显示的是上一交易日的数字，
   叫「今日」名不副实；「最新」在所有时段都准确，且不需要按状态切换标题。
4. **红线 #6 由「三处逐字一致」改为「两类口径」**（见下）。

### 理由

- **为什么闸门放在"是不是今天"而不是"有没有估算"**：用户原话是「有实时估算数据的才参与计算，
  官方优先于估算」。但盘后官方出来时估算也在，此时该用官方；而盘中官方永远是昨天的，该用估算。
  两种情况的共同点不是"有没有估算"，是**"这份数据是不是今天的"**。按日期闸门，两种情况自然都对。
- **为什么必须同时改抽屉的 profitMap**：若只改顶部，就会精确复现 [D-010](#d-010) 当初修掉的那个
  bug——「顶部今日涨跌归零，抽屉里却照常有收益，两者对不上」。**顶部和抽屉是同一个口径的两个出口，
  必须同步。**
- **为什么 `getNavByCode` 不加闸门**：它服务的是**市值**（持仓值多少钱、权益百分比），
  该用最新可得净值，哪天的都对。加闸门反而会让持仓总额在盘中归零。
  **这就是为什么"三处逐字一致"这条红线从一开始就不该是三处——它把两种口径捆在了一起。**
- **为什么非交易日不显示 `-`**：周末看最近一个交易日的结果是有价值的，且周末没有"今天该有数据"
  这回事，不构成异常。改名之后也不再有名实不符的问题。
- **为什么不做节假日日历**：`getMarketState` 只判周末，不判法定节假日。国庆等长假期间会被当作交易日，
  因而显示 `-` 而非最近收益。**已知限制，记入 05，不硬造日历**——维护一份年年要更新的节假日表，
  代价大于它解决的问题。

### 代价

- 红线 #6 的表述必须改，CLAUDE.md 与 `docs/03` §3.3 同步。**这是本次唯一动到红线的改动。**
- 法定节假日期间「最新收益」显示 `-`（见上，已知限制）。
- 盘中估算断供时用户看不到任何收益数字。**这正是目的**——没有今天的数据就不编今天的数字，
  与 [D-001](#d-001)、[D-018](#d-018) 同向。

### 代码位置

- `js/engine.js` `calcTodayProfit`：日期闸门 + 非交易日回退
- `js/interact.js` `openHoldingDrawer` 的 `profitMap`：同口径同步改
- `js/data.js` `getNavByCode`：**不动**（市值口径）
- `index.html`：`今日收益` / `今日涨跌` → `最新收益`
- `js/ui-holding.js`：抽屉表头 `今日收益(¥)` → `最新收益(¥)`

### 须同步修订的正文章节

- `CLAUDE.md` 红线 #6（三处逐字一致 → 两类口径）
- `docs/03-实现约束.md` §3.3（净值判据表拆成市值/收益两类）
- `docs/05-未实现项.md`（法定节假日日历，已知不做）

### 取代关系

不取代 [D-010](#d-010)——徽标与数值分家这个结构仍然成立，且本条正是靠它才定位到问题。
**收紧的是 D-010「数值可以放宽（fallback）」那半句**：fallback 的范围从"任意可得数据"
缩小为"今日数据，官方优先"。D-010 状态保持「生效中」。

## D-018 · 估算缓存按数据自带时间戳标记陈旧；Gist 推送时机与读侧节流修正

**状态**：生效中
**日期**：2026-07-21
**关联**：[D-014](#d-014)（三层回退，本条修正其实现）、[D-001](#d-001)（陈旧必须可见）、[D-015](#d-015)

### 背景

D-014 定的三层回退（fresh → localStorage → Gist → unavailable）**结构正确**，但实现有三处与其描述不符，2026-07-21 全量审查发现：

1. **陈旧标记根本不存在。** D-015 的理由栏白纸黑字写着「估算列会显示陈旧值（打"陈旧"标记）」，
   但 `js/data.js` 回退时返回 `source:"cached"/"gist"`，而 `js/ui.js` 的 `srcTag()` 只认
   `"backup"/"stale"`，这两个值落进去返回空串。**用户看到的是：缓存的旧估算与实时数据外观完全一致，
   零提示。** 这正是 [D-001](#d-001) 事故文档所禁止的「静默沿用」——上次是错误清空，这次是无声沿用，
   同一条原则的两个反面。
2. **Gist 兜底会把旧日期数据盖成今天。** `_lsSaveEstCache` 无条件写 `date: todayDateStr()`，
   而 Gist 取回路径直接调它落盘 → 一份隔周的快照被标成「今日数据」，TTL 从 2h 变 18h，
   配合第 1 点用户完全无从察觉。
3. **推的是当日第一笔，不是收盘快照。** `_maybePushEstToGist` 挂在每次成功之后，且当日标记
   **先于**推送写入 → 实际推的是 09:31 开盘估算，且推送失败当天不再重试。D-014 原意是「收盘后推送一次」。

**连带发现**：因第 3 点，`fm_est.json` 从未被真正写入 → 每次刷新都走 Gist 分支 → 404 → 异常被吞。
**每 60 秒一次带 token 的 GitHub 认证请求，全部无效**，且串在 `refreshData` 的 await 链上。
D-015 记的「白等一次超时」只覆盖了行情请求，没算这一条。

### 决策

1. 缓存结构去掉 `date` 字段，只留 `{ ts, data }`；`ts` 仅作硬上限（`EST_CACHE_MAX_AGE`，7 天）整份丢弃。
2. **新旧判断改由每个条目自带的 `estimateAt` 决定**。`srcTag` 扩展为按数据日期打标签：
   `estimateAt` 的日期 ≠ 今天 → 「陈旧」；是今天 → 不打（那确实是今日最后一笔，且时间已显示在旁边）。
3. Gist 推送时机移出 data 层，由 `js/interact.js` 的 `refreshData` 在 `mkt === "POST_MARKET"` 时触发；
   当日标记改为**推送成功后**才落。
4. Gist 读侧加节流（`EST_GIST_READ_THROTTLE`，30 分钟），**无论成败都推进时间戳**（失败即负缓存）。
5. 缓存读写移入 store 封装，走既有的 `_lsGet`/`_lsSet` 安全外壳。

### 理由

- **为什么按数据自带时间戳、而不按来源打标签**：来源只说明「这次是从哪拿的」，说明不了「数据本身多旧」。
  按来源打，收盘后沿用当日 14:59 那笔也会被标成陈旧（不准确）；按数据日期打，标签永远反映真实新旧，
  **并且顺带让第 2 点那个 bug 无处可藏——没有 `date` 字段可以被盖。** 修根源而不是加判断。
- **为什么推送时机放 interact 而不是 data**：`getMarketState` 属 engine(4)，而 data(3) 与 engine
  **平级互不依赖**（[02-系统架构.md](02-系统架构.md) §2.2）。interact(6) 是唯一能同时看到两边的合规位置，
  也让 data 层回归「只管取数与缓存」。
- **为什么收盘后第一个 tick 就是对的时机**：`refreshData` 每 60 秒一次，收盘后首个 tick 落在
  15:00–15:01，此时缓存里正是当日最后一笔（14:59）——既满足「收盘后」，也满足「最新」。
- **为什么读侧失败也要推进时间戳**：`fm_est.json` 可能长期不存在（估算源断供时永远推不上去）。
  只在成功时节流等于没节流。
- **D-014 状态保持「生效中」**：三层回退结构没有被推翻，本条修的是实现与其描述不符之处，
  不是决策替换。按只增不改的规矩，不动 D-014 原文。

### 代价

- `js/config.js` 增三个常量，`js/store.js` 增两个封装，`js/data.js` 净减（删三个私有函数 + 一个死变量）。
- 收盘前若浏览器一直没打开，当天不推 Gist。可接受：Gist 只服务「换设备冷启动」，
  而盘中冷启动本来就能直接取到 fresh 数据。

### 代码位置

- `js/config.js`：`STORE_EST_GIST_DATE`、`EST_CACHE_MAX_AGE`、`EST_GIST_READ_THROTTLE`
- `js/store.js`：`saveEstCache` / `loadEstCache`
- `js/data.js`：删 `_lsEstKey` / `_lsSaveEstCache` / `_lsLoadEstCache` / `_estGistPushedDate`；
  `_maybePushEstToGist` → `pushEstToGist`；`fetchEstimates` 读侧节流
- `js/ui.js`：`srcTag(source, dataDate, today)`；调用点在 `buildCardInnerHtml`
  **与 `buildTableInnerHtml`**——后者原本从不调 `srcTag`，导致桌面表格视图连既有的「备用」
  标签都从未出现过。只补卡片视图等于在 PC 上仍然静默沿用，D-018 的可见性承诺会架空一半。
- `js/interact.js`：`refreshData` 的 `POST_MARKET` 推送门

### 须同步修订的正文章节

- `docs/03-实现约束.md` §3.3（估算持久化段）
- `docs/02-系统架构.md` §2.4（直连模式估算说明）
- `docs/05-未实现项.md`（D-015 条下补指向）

**「不可用」和「静默沿用」都必须是对用户可见的状态。**

## D-017 · PE 锚定路径由 `PE_ANCHOR` 统一，持仓抽屉与 PE 栏对齐

**状态**：生效中
**日期**：2026-07-21
**关联**：[D-006](#d-006)（1.0/2.0 精度实测）、[D-013](#d-013)（`DATA_MODE` 同款常量开关惯例）

### 背景

`isEquityWrongDir`（方向异常判断）在两处用了**不同的 PE 源**：

| 位置 | 传入的 PE | 路径 |
| --- | --- | --- |
| `js/ui-pe.js` `updatePeBar` | `getEnginePE1` | 1.0 总市值路 |
| `js/interact.js` `openHoldingDrawer` → `js/ui-holding.js` | `getEnginePE` | 2.0 点位路 |

而 `js/engine.js` 注释与 [02-系统架构.md](02-系统架构.md) §2.6 都写明 1.0 是「PE 主路径
（大号 PE / bar / 触发状态 / **权益判断**）」。[D-006](#d-006) 实测：1.0 平均绝对误差 0.41pp / 最大 0.86pp；
2.0 平均 1.31pp / 最大 3.69pp。

**用户会看到**：PE 栏的权益数字染成正常色（1.0 判定不异常），点开持仓抽屉，同一时刻同一份持仓却染成
`--warn` 橙色（2.0 判定异常）——或者反过来。`PE_HIGH_THRESHOLD` 是 65，实时 PE 越贴近这条线越容易撞上。

### 决策

新增 config 常量 `PE_ANCHOR`（默认 `"mcap"`），engine 新增 `getAnchorPE` / `getRefPE` 两个分发纯函数，
**全站 PE 锚定收口到 `getAnchorPE`**；PE 栏小字改用 `getRefPE`，永远展示另一条路。

### 理由

- **为什么不直接把抽屉那处改成 `getEnginePE1` 了事**：那能修掉当下的 bug，但「哪条是主路径」这个知识
  仍然只散落在注释里，下次新增调用点还会再分叉一次。收口到一个常量 + 一个分发函数，
  **分叉在结构上就不可能再发生**。
- **为什么保留可切换而不写死 `"mcap"`**：[D-006](#d-006) 明确记了 2.0 的保留理由——
  「成分调整日总市值会跳变，那种日子点位路反而是有效参照」。写死等于删掉这个逃生口。
- **为什么沿用常量开关这个形态**：`DATA_MODE`（[D-013](#d-013)）已经是同一套惯例——一个常量、
  全站切换、改动收口在分发点。**项目里已有的习惯优先于新发明。**

## D-014 · 估算数据持久化：localStorage + Gist 三层回退

**状态**：生效中
**日期**：2026-07-20
**关联**：[D-012](#d-012)（直连模式例外）、[D-013](#d-013)（直连数据源）

### 背景

直连模式（D-013）下，腾讯行情盘后估算字段返回 0 → `_parseTencentFunds` 的 `estimateNav > 0` 过滤
→ 估值列全部变成 `"--"`。D-012 为了网关性能加了盘后跳过估算的优化，但在直连模式不需要
（与官方链共享 in-flight 去重，无额外网络代价），反而造成了"收盘=空白"的体验问题。

更严重的是：即使当时显示了，换设备或清缓存后也全部丢失。in-memory 变量不能跨设备持久化。

### 决策

估算数据做三层回退持久化：

```
fresh（腾讯直连有数据 → 返回）
  ↓ 空
localStorage（本地缓存 → 返回，标记来源 "cached"）
  ↓ 空
Gist fm_est.json（跨设备兜底 → 返回 + 同步到 localStorage，标记来源 "gist"）
  ↓ 空
unavailable（真正没数据）
```

### 理由

- **localStorage**：即时、无网络、解决同设备刷新/关机后恢复。当日数据 TTL 18h。
- **Gist**：复用现有 Gist 同步基础设施（`_cloudReadFile`/`_cloudWriteFile`），新文件 `fm_est.json`。
  收盘后推送一次（fire-and-forget，每天一次），解决换设备首次打开的问题。
- **代理模式不影响**：`DATA_MODE="gateway"` 时 `needEstimate` 策略原样保留。

### 代价

- `js/data.js` 新增约 50 行（三个缓存函数 + Gist 推送）。
- 新增 `GIST_FILE_EST = "fm_est.json"` 到 `config.js`，一个 localStorage key `STORE_EST_CACHE`。
- `interact.js` 的 `needEstimate` 加了一条 DATA_MODE 分支判断。

### 代码位置

- `js/config.js`：`GIST_FILE_EST`、`STORE_EST_CACHE`
- `js/data.js`：`_lsSaveEstCache` / `_lsLoadEstCache`、`_maybePushEstToGist`、`cloudReadEst` / `cloudUpdateEst`；
  `fetchEstimates` 直连模式改为 async，内联三层回退
- `js/interact.js`：`needEstimate` 在 `DATA_MODE="direct"` 时为 `true`（始终调 `fetchEstimates` 走缓存）

## D-013 · 数据源双模：`DATA_MODE` 切换网关 / 腾讯直连

**状态**：生效中
**日期**：2026-07-20
**取代**：[D-008](#d-008)（"前端一律不直连第三方行情"）—— 直连模式即前端直连腾讯，故 D-008 不再作为全局红线，降级为"网关模式下的约束"。

### 背景

原架构（D-008）前端只走 Cloudflare 网关 `API_BASE`，主备切换 + KV last-known-good 全在网关侧。
迁移后实测发现：**Cloudflare Worker 出站 IP 在境外，访问国内数据源天然慢 4–8 倍**
（东方官方 1.3s vs 直连 0.28s；腾讯行情 0.6s vs 直连 0.15s）。
而天天基金 `fundgz` 老接口已 301→404 死亡，东方 `FundMNFInfo` 前端直连被 `ErrCode:61136` 拦截，
**唯一还能前端直连的是腾讯 `qt.gtimg.cn`**（带 `Access-Control-Allow-Origin: *`，GBK 编码）。

用户怀念早期前端直抓的丝滑感，且明确"不强制主备"，故选定路线 A：前端直连腾讯。
同时为保留网关作为兜底，不删除 `workers/` 代码，而是用**一个开关**让两条链路并存。

### 决策

1. **`config.js` 加 `DATA_MODE` 常量**（`"gateway"` | `"direct"`），当前默认 `"gateway"`。
   改这一个常量 + 推一次即全站切换，不动任何业务代码。
2. **`DATA_MODE === "direct"` 时**：`js/data.js` 走 `_fetchTencentFunds` / `_fetchIndexGroupTencent`，
   浏览器直连 `TX_BASE`，字段产物与网关**逐字段对齐**，保证 `fetchSingleFund` / `setIndices` / `setQQIndex` 无感切换。
3. **`DATA_MODE === "gateway"` 时**：完全沿用既有网关路径，行为不变。
4. **`workers/` 网关代码原样保留**，不删。直连模式前端不请求它，仅作备选。

### 理由

- **为什么用运行时开关而非两条 Git 分支**：线上静态站只认一条分支，切分支要重新合并部署，
  且两套 `data.js` 长期并行必冲突。一个常量 + 一份代码同时装两套数据源，切换成本最低、回滚最稳。
- **为什么只用腾讯单源**：fundgz 已死、东方前端被 61136 拦，仅腾讯可直连。单源 = 无主备，
  用户已确认接受（直连模式无 last-known-good 兜底，断网即无数据；如需可后续加 localStorage 缓存）。
- **为什么字段要对齐网关**：渲染/计算层只认 `estimateNav/officialNav/f2/f3/...` 这套字段名，
  直连产物保持同构，下游零改动。

### 代价

- **代码量增加**：`js/data.js` 新增 TX 直连段（约 130 行），临时突破红线 #8"改后代码量不许增加"。
  该红线就本特性**临时放行**——双链路是用户明确要的功能，无法在不增代码的前提下实现。
- **直连模式无 LKG 兜底**：网关的 KV last-known-good 在直连模式失效；断网/腾讯故障 = 数据空白。
- **GBK 解码硬依赖**：必须用 `TextDecoder("gbk")`，浏览器原生支持；若某环境不支持该标签会乱码。
- **恒生指数字段差异**：`hkHSI` 布局比 A 股指数短（78 vs 88 字段），其 PE/市值/时间为 0/null，
  与网关同字段位置表现一致；显示与 PE 锚定（锚定沪深300）不受影响。

### 代码位置

- `js/config.js`：`DATA_MODE`、`TX_BASE`、`TX_INDEX_QQ`
- `js/data.js`：`_txNum` / `_txDate` / `_parseTxAssignments` / `_fetchTencentFunds` / `_parseTencentFunds` / `_fetchIndexGroupTencent`；
  `fetchOfficialData` / `fetchEstimates` / `_fetchIndexGroup` 三处按 `DATA_MODE` 分流

## D-011 · 网关 in-flight 请求去重

**状态**：生效中
**日期**：2026-07-20

### 背景

`router.mjs` 的 `cache` Map 只缓存"已完成结果"，不缓存"进行中的 Promise"。
同一 cacheKey 在 TTL 窗口内并发到达时，所有请求都各自调起 `selectGroup` → 打上游，
直到第一个完成写回缓存。触发条件窄（TTL 8s~60s 窗口内并发重叠），但一旦触发就是
无意义的重复上游调用。

### 决策

加 `inflight` Map，存储进行中的 Promise。请求到达时：
1. 有 inflight → 搭车等同一个 Promise
2. 有已完成缓存 → 秒回
3. 都没有 → 发起一趟，写进 inflight，`finally` 中释放

**`force` 参数跳过去重**（诊断用）。

### 理由

- 对单机自用影响极小，但对 PWA 公用部署或爬虫误触有防护价值。
- 改动 <15 行，风险低。
- Cloudflare Pages Functions 是 per-isolate，跨 isolate 不共享 inflight，
  同 isolate 内的幂等已是当前能达到的最大保护。

### 代码位置

`workers/fund-market-api/src/router.mjs`

<a id="d-010"></a>
## D-010 · "已更新"徽标与净值选取判据拆开

**状态**：生效中
**日期**：2026-07-20

### 背景

`calcTodayProfit` 中 `isOfficialUpdated` 身兼两职：
1. 决定 nav/pct 取哪边（官方还是估算）
2. 驱动 `allUpdated` → 顶部"已更新"徽标

2026-07-20 晚间盘中估算整组不可用（`status: "unavailable"`），`estD` 为空，
`!estD` 短路使整条判据退化为 `!!f.offVal`——任何有官方净值的基金都被判为"已更新"，
与 `offD` 是不是今天完全脱钩。用户看到部分基金官方还是昨天的，徽标却亮着。

### 决策

**净值选取口径（`isOfficialUpdated`）保持不变**：`mktState === "WEEKEND" || "BEFORE_PRE" || !!(offVal && (!estD || offD >= estD))`。
三处（`getNavByCode`、`openHoldingDrawer` 的 profitMap、`calcTodayProfit` 的 nav/pct 选取）维持一致。

**新增"徽标口径"（`isFundUpdated`）**，仅用于驱动 `allUpdated`：
```javascript
const isOffToday = offD === todayStr;
const isEstToday = estD === todayStr;
const isFundUpdated = isOffToday || isEstToday;
```

**两者分工**：`isOfficialUpdated` 管数值（放宽 fallback，避免整只跳过导致今日涨跌归零）；
`isFundUpdated` 管状态（严格，必须有今天的数据才算"已更新"）。

### 理由

- **数值可以放宽、状态必须严格**：估算空时用昨天官方凑出 0 涨跌是可接受的 fallback；
  用昨天数据显示"已更新"会误导用户，比显示空白更危险。
- **估算整组不可用是常态（盘后）**，不能因此把"放宽的 fallback"误用到状态判定上。

### 代码位置

`js/engine.js` `calcTodayProfit`

### 教训

**判定状态徽标 vs. 判定数值选取，是两件不同的事。** 未来加任何带"状态标签"的 UI 元素，
先问自己："这条展示信息的前提判据，和数值选取的判据，是同一道题吗？"

<a id="d-001"></a>
## D-001 · 行情数据 last-known-good 保护，落在网关侧

**状态**：生效中
**日期**：2026-06-19 确立（前端版）→ 2026-07-20 被拆除（事故）→ 2026-07-20 重建于网关侧

### 背景（含事故经过）

2026-06-19 的 `9a41e0e`（298 行，横跨 data/store/ui）确立了一条原则：
**行情数据拿不到新的，就继续用上次的好数据，并明确标记为陈旧**。当时的实现是
`setIndicesUnavailable()` → 回退 localStorage 快照 → `mode:"stale"` → UI 上 `is-stale` 灰化。

这条原则**从未写进任何文档**，只活在指数那条代码路径里。

2026-07-20 做网关迁移（`b76b928`）时，AI 读到文档里 2.6 节"任一组请求失败即整组降级为不可用"
和"`unavailable` 时官方字段保持空值"，照着执行，于是：

- `data.js` 官方组失败时 `delete officialBatchCache[cacheKey]`——**主动删掉**上次的好数据。
  旧实现 `if (result) offCache[code] = ...` 是失败**不写**缓存，下次 TTL 内继续返回上次好数据。
- 失败波及范围从"每只独立"（`funds.map(fetchSingleFund)`）变成"一只解不出、全组归零"
  （网关 `records.every(Boolean) ? records : null`）。
- `setLastResults(res)` 整体覆盖，没有指数那样的 stale 回退。

**用户看到的现象**：收盘后天天基金/腾讯都停供盘中估算，整组 unavailable，
一次 60 秒刷新就把一整天的估算数据冲成 `--`。

### 决策

1. **行情数据禁止因单次失败清空。** 拿不到新数据 → 返回上次好数据 + 标记陈旧。这是硬红线。
2. **保护落在网关侧，不落在浏览器。** 网关持久化最后一份完整好数据（KV），
   失败时以 `status:"stale"` 返回。前端只负责把陈旧态显示出来。
3. 前端原有的指数 localStorage 快照保留，作为"网关整个够不着"时的第二道防线。

### 理由

- **为什么不能只在前端**：localStorage 是**单机**的。换一台电脑、或手机冷启动，
  晚上打开看板照样是空白——保护等于没有。网关侧存一份，所有设备一次受益。
- **为什么是 KV 而不是 `router.mjs` 里那个 `cache` Map**：那个 Map 活在 isolate 内存里，
  冷启动即丢，跨 colo 不共享，撑不住"晚上打开还能看到当天数据"。
- **为什么必须标记陈旧而不是静默沿用**：看板是盯盘工具，用户据此判断要不要下单。
  把昨天的数当今天的显示，比显示空白更危险。**陈旧可以接受，陈旧且不告知不行。**

### 代价 / 已知副作用

- KV 有写入配额，需对写入节流（不是每次成功都写）。
- 陈旧数据的"保质期"需要定上限，跨交易日的旧数据不应继续冒充。
- `status` 多出 `stale` 一态，网关契约、前端 `_fetchGroup`、测试、README 都要跟着改。

### 代码位置

- 网关：`workers/fund-market-api/src/router.mjs`、`src/lkg.mjs`
- 前端消费：`js/data.js` `_fetchGroup`；前端指数第二道防线：`js/store.js` `setIndicesUnavailable`
- 陈旧态呈现约定：`js/ui-pe.js` `is-stale` / `js/ui.js` `.stale`

### 教训（这条比决策本身更重要）

**原则只写在代码里，等于没写。** 更糟的是，文档里当时还留着一句看起来相反的话（2.6 的"保持空值"），
于是 AI 越是老实照文档执行，越是精准地把保护拆掉。
`docs/` 的归档与本日志机制就是为堵这个洞而建，见 `CLAUDE.md` 第三节。
